# ADR: AI Query Entrypoint and SBLR Policy

Date: 2026-02-18
Status: Accepted

## Context

ScratchBird architecture enforces parser isolation and SBLR execution boundaries:

- Engine executes verified SBLR, not raw SQL.
- Parsers are untrusted clients and must handle SQL->SBLR compilation.
- Engine remains parser-agnostic and enforces authorization/execution semantics.

AI integration must choose where generated queries enter the system.

## Decision

Use parser/compiler entrypoints as the default AI query path.

- AI systems generate dialect/native query text.
- Query text is compiled via existing parser/compiler pathways into SBLR.
- Engine executes validated SBLR.

SBLR-direct generation by AI is disallowed in baseline scope.

Optional later scope:

- Allow only controlled SBLR template/macro invocation via approved compiler wrappers.
- No free-form model-generated bytecode submission.

## Consequences

Positive:

- Preserves ScratchBird’s core parser/SBLR boundary invariants.
- Reuses existing dialect behavior and compatibility paths.
- Reduces risk of invalid/unsafe bytecode construction by LLM output.

Negative:

- Adds parser dependency for all SQL-like AI workflows.
- May add latency versus direct bytecode authoring.

- Define compile/execute interface contract in `../drafts/TEXT_TO_SQL_ROUTER_AND_COMPILER_SPEC.md`.
- Add policy checks rejecting raw bytecode authoring requests from model outputs.
